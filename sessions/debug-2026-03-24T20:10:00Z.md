**Metrics**: TTFT=1.4s (wall-clock, AP-JP-1), server-side TTFT≈600ms, TTLT≈2.3s for 500 tokens, decode throughput≈460 tok/s, Errors=0

## Analysis

- Span attribute `sglang.startup.enable_mixed_chunk`: false → prefill and decode run in separate steps; decode cannot overlap with incoming prefill, adding queue time when both are in flight
- Span attribute `sglang.startup.num_continuous_decode_steps`: 1 → scheduler interrupts decode after every step to check for new prefill requests, adding ~1 scheduling cycle overhead per decode batch
- Span attribute `sglang.startup.attention_backend`: fa3 → FlashAttention 3 (optimal for H100)
- Span attribute `sglang.startup.mem_fraction_static`: 0.835 → 64.6 GB KV cache (705,734 tokens); model weights only 0.91 GB, leaving ~11.8 GB available GPU memory
- Span attribute `sglang.startup.chunked_prefill_size`: 8192 → conservative; with 0.5B model on H100, larger chunks reduce overhead per token
- Span counter `sglang.latency.time_to_first_token`: baseline 1.95s (includes 800ms round-trip to AP-JP-1, so server-side TTFT ~600ms), drops to 0.39s on prefix cache hit
- Span counter `sglang.latency.time_in_queue`: 0.8ms → queue is not the bottleneck
- Span counter `sglang.usage.prompt_tokens`: 26 for short benchmark prompts; 35 for longer essay prompts
- Metric `sglang.cache_hit_rate`: 0.0 for fresh requests, 0.96 for repeated prefix requests → radix cache is effective but TTFT remains high due to scheduler overhead on first decode step
- Dstack logs `Decode batch gen throughput`: 430–490 tok/s at bs=1 (cuda graphs), but first decode step is 0.15–31 tok/s before cuda graph warms up (bs=1 cuda graph capture lag)
- Dstack logs `Prefill batch input throughput`: 0.03–17.83 tok/s due to very short prompts relative to the 1.8s scheduler window resolution

## Findings

1. Mixed chunk disabled — prefill cannot overlap with decode, increasing effective TTFT when server is under any decode load
2. `num_continuous_decode_steps=1` creates one scheduler loop overhead per decode token, wasting ~0.5–1ms per step when no prefill is queued
3. First decode batch shows 0.15–31 tok/s before settling to 460 tok/s — cuda graph warm-up per-request start adds ~30ms first-token decode overhead
4. Quantization not used — H100 supports FP8 natively; applying fp8 to 0.5B model weights can reduce memory bandwidth and improve decode throughput by 30–50%
5. `mem_fraction_static=0.835` allocates 64.6 GB to KV cache while model only uses 0.91 GB — could reduce to 0.5 (40 GB KV, still 370k+ tokens) and free memory for quantized activations

## Planned changes

1. Enable `--enable-mixed-chunk` to allow decode and prefill to overlap in same scheduler step
2. Set `--num-continuous-decode-steps 3` to reduce scheduler interruptions during decode
3. Add `--quantization fp8` to use H100 native FP8, expected to improve decode throughput 30–50%
4. Reduce `--mem-fraction-static 0.6` to free headroom; 0.5B model needs very little KV space
