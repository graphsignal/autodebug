## Findings

1. FP8 weight quantization active but decode throughput unchanged — weight memory bandwidth is NOT the bottleneck for this 0.5B model; KV cache BW and per-step overhead dominate
2. `kv_cache_dtype=bfloat16` still consuming full memory bandwidth for KV reads/writes — applying FP8 to KV cache could reduce KV bandwidth by 2x
3. Event loop overhead eliminated by `num_continuous_decode_steps=5` — scheduler contribution to latency now negligible (~93 event loop passes/s)
4. No speculative decoding configured — ngram speculative decoding requires no draft model and could propose 3 tokens at once, verified in one forward pass; expected 1.5–2.5x throughput for repetitive content
5. First decode step warm-up overhead (~25ms) suggests cuda graph launch latency adds to per-request TTFT; already using cuda graphs but first-batch scheduling has extra overhead
6. Fresh TTFT improved 1.49s → 1.16s (22%); likely from `enable_mixed_chunk` reducing scheduling overhead when server is transitioning from idle to decode
