## Analysis

- Span attribute `sglang.startup.enable_mixed_chunk`: false → scheduler cannot overlap prefill and decode steps; new requests wait for current decode batch to complete its step before prefill can be scheduled
- Span attribute `sglang.startup.num_continuous_decode_steps`: 1 → scheduler returns to event loop after every decode step to check for incoming requests; ~38µs scheduling overhead per step (from `Scheduler.event_loop_overlap` avg ~38µs/iteration), adding ~20ms per 500-token response
- Span attribute `sglang.startup.attention_backend`: fa3 → FlashAttention 3 (optimal for H100 SXM)
- Span attribute `sglang.startup.mem_fraction_static`: 0.835 → 705,620 token KV capacity; model weights = 0.91 GB bfloat16, so vast majority of 64.6 GB KV allocation is unused for this 0.5B model
- Span attribute `sglang.startup.chunked_prefill_size`: 8192
- Profile `sglang.srt.model_executor.model_runner.ModelRunner.forward_extend`: avg 3.6–7.2 ms/call → prefill of 26-token prompt takes ~4ms on H100; not the bottleneck
- Profile `sglang.srt.managers.scheduler.Scheduler.run_batch`: peak 511 calls/bucket → dense decode; consistent ~38µs per event loop pass
- Metric `sglang.e2e_request_latency_seconds`: 2.056s first request (no cache), 0.534s after prefix cache warm-up → cache hit rate drives TTLT from 2.0s → 0.53s
- Metric `sglang.inter_token_latency_seconds`: 0.0018–0.0027s → single-request decode 370–555 tok/s; batch decode 2,802 tok/s at bs=7 (from dstack logs)
- Dstack logs `Decode batch gen throughput`: 500–555 tok/s at bs=1, peaks at 2,802 tok/s at bs=7 with cuda graphs; strong batch scaling
- Span counter `sglang.usage.cached_tokens`: 312 tokens served from radix cache during parallel benchmark → prefix sharing effective

## Findings

1. No quantization used — model is BF16, 0.91 GB weights; H100 supports FP8 natively; FP8 would halve weight memory bandwidth pressure, expected to improve bs=1 decode throughput from ~540 → ~800+ tok/s
2. `enable_mixed_chunk=False` — for single sequential requests this has minimal impact, but under concurrent load (8 parallel) TTFT is 1.62s because new prefills must wait for decode step to finish first
3. `num_continuous_decode_steps=1` — scheduler overhead accumulates: ~38µs × 500 steps = ~19ms extra overhead per long response; increasing to 5 saves ~15ms decode phase
4. `mem_fraction_static=0.835` — 64.6 GB KV for a 0.5B model is overkill; reducing to 0.4 still gives 300k+ tokens, frees ~20 GB
5. First-batch decode warm-up: first decode step at bs=1 runs at 50–95 tok/s before cuda graph kicks in, adding ~30ms to first-token decode time

## Planned changes

1. Add `--quantization fp8` — H100 native FP8, target 30–50% decode throughput improvement
2. Add `--enable-mixed-chunk` — overlap prefill and decode for lower TTFT under concurrent load
3. Set `--num-continuous-decode-steps 5` — reduce scheduling overhead per decode step
4. Reduce `--mem-fraction-static 0.5` — still 350k+ token KV capacity, frees 27 GB
